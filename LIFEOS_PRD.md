## 1. Project overview

### Problem
Two interconnected problems to solve simultaneously:

**Macro (drift):** Multiple active projects competing for limited evening hours with no prioritisation framework. Effort is diffuse, progress is inconsistent.

**Micro (chaos):** Days lack basic structure — mealtimes, sleep, work blocks, and household tasks are all reactive. The life infrastructure that sustained project work depends on does not exist yet.

### Solution
A single mobile-first daily-driver app that connects day-to-day structure (habits, routines, life tasks) to long-term goals (areas, rocks, north star). Every action traceable to the life being built.

### North star
> Full-time digital nomad swing trader by Q2 2028

### Active project areas at launch

| Area | Type | Focus budget |
|---|---|---|
| Trading | Income-building | 35% |
| Trading journal app | Project | 20% |
| Life OS | Project | 15% |
| Peak Processing (day job) | Obligation | 20% |
| Financial / lifestyle goals | Vision | 10% |

### User profile
- Single user — no multi-tenancy required
- 75% mobile / 25% desktop usage split
- Mobile: frequent throughout the day (check, log, quick add)
- Desktop: evenings for deeper work sessions
- Starting from zero routine — system must not feel punitive

## 2. Tech stack

| Layer | Technology | Notes |
|---|---|---|
| Frontend | React + TypeScript | Vite for bundling |
| Styling | Tailwind CSS | Mobile-first, v3 |
| Data fetching | TanStack Query (React Query v5) | Caching, background refetch |
| Routing | React Router v6 | |
| Backend | Node.js + Express | TypeScript |
| Database | PostgreSQL | Hosted on Render |
| ORM | Drizzle ORM | Type-safe, pairs well with TS |
| Auth | JWT | Single user, username + password |
| Frontend hosting | Vercel | Auto-deploy from main branch |
| Backend hosting | Render | Web service + managed Postgres |
| PWA | Vite PWA plugin (vite-plugin-pwa) | Workbox under the hood |

### Why these choices
- Drizzle ORM preferred over Prisma — lighter, better raw SQL escape hatch, schema-as-code fits a solo project
- TanStack Query handles optimistic updates for the tick-off interactions without manual cache management
- Vite PWA plugin generates service worker and manifest automatically — PWA is day one, not bolt-on
- Same backend pattern as the existing trading journal — consistent across projects

### 4.2 Schedule config spec

The `schedule_config` JSON field on `life_tasks` uses the following shapes:

```typescript
// schedule_type: 'daily'
{}

// schedule_type: 'days_of_week'
{ days: ('monday' | 'tuesday' | 'wednesday' | 'thursday' | 'friday' | 'saturday' | 'sunday')[] }
// e.g. { days: ['wednesday'] }
// e.g. { days: ['monday', 'wednesday', 'friday'] }

// schedule_type: 'every_n_weeks'
{ every: number, day: string }
// e.g. { every: 2, day: 'sunday' }

// schedule_type: 'monthly_date'
{ date: number }
// e.g. { date: 1 }  → 1st of the month

// schedule_type: 'monthly_occurrence'
{ occurrence: 'first' | 'second' | 'third' | 'fourth' | 'last', day: string }
// e.g. { occurrence: 'first', day: 'monday' }

// schedule_type: 'weekly_floating'
{}  // appears in "this week" section, flexible on day
```

### 4.3 Schedule engine (`src/utils/scheduleEngine.ts`)

The schedule engine resolves which `life_tasks` are due on a given date. Called on each `GET /api/life-tasks/today` request.

```typescript
// Function signature
function getTasksDueOn(tasks: LifeTask[], date: Date): LifeTask[]

// Logic per schedule_type:
// 'daily'              → always due
// 'days_of_week'       → due if date.dayOfWeek in config.days
// 'every_n_weeks'      → due if weeks since task.createdAt % config.every === 0
//                        AND date.dayOfWeek === config.day
// 'monthly_date'       → due if date.date === config.date
// 'monthly_occurrence' → due if date matches e.g. first Monday of month
// 'weekly_floating'    → always included in weekly list, not daily list
```

## 7. Feature specifications

### 7.1 Habits

- Circular checkbox (visually distinct from life tasks square)
- Streak calculated from consecutive days of `completed: true` in `habit_logs`
- Streak displayed as "N day streak" next to habit name
- Nudge logic (see section 9)
- Default habits at seed:
  1. Pre-market chart review (linked to Trading area)
  2. Trade journal entry logged (linked to Trading area)
  3. Life OS session log completed (linked to Life OS area)

### 7.2 Life tasks — three types

**Recurring scheduled:** Task definition in `life_tasks`. Schedule engine determines if due today. Completion logged in `life_task_logs` with `taskId` populated and `isOneoff: false`.

**Weekly floating:** `scope: 'weekly'`, `scheduleType: 'weekly_floating'`. Appears in weekly section of Today screen. Persists until ticked this week.

**One-off day tasks:** No task definition. Logged directly to `life_task_logs` with `taskId: null`, `isOneoff: true`, `note: 'Sort the shelf unit'`. Created via the inline add row at the bottom of the life tasks card.

### 7.3 Routines (weekly pattern view)

Four default routines tracked in `daily_logs`:
1. Wake time (logged as `wakeTime` — time field)
2. In bed by target (logged as `sleepTarget`)
3. Cooked dinner (boolean `cookedDinner`)
4. Work block done (boolean `workBlockDone`)

Routines card on Today screen shows 7 dots for the current week (Mon–Sun), coloured:
- Green dot: logged and completed / on target
- Amber dot: logged, slightly off target (e.g. 30 min past sleep target)
- Red dot: explicitly logged as not done
- Grey dot: no log for that day yet

### 7.4 Weekly review

Review form lives in the Review tab. On opening, checks if this week's review exists — if not, presents the form. If it does, shows the completed review with an Edit option.

Review form sections:
1. Area momentum check (read-only — shows session count per area this week)
2. Focus allocation (read-only — actual vs intended budget)
3. Routine pattern (read-only — the week's dot patterns)
4. Wins this week (free text)
5. What got in the way (free text)
6. Next week's priority area (single select from active areas)
7. Rock progress (status per active rock — on track / at risk / done)

### 7.5 Project boards

Per-area project board accessed from the area detail screen.

Four columns rendered as vertically stacked sections on mobile:
1. **Next actions** — ordered list. Top item is the "next action" shown in the area cockpit card.
2. **In progress** — hard cap of 3 items enforced on creation. Warning shown if attempting to add a 4th.
3. **Decisions needed** — items blocking progress.
4. **Done this quarter** — completed items, most recent first.

Note/reference items rendered separately below the board.

### 7.6 Focus allocation (insights)

Sessions are the unit of measurement. Focus allocation = sessions per area as a percentage of total sessions in the period.

`GET /api/insights/focus?from=2026-03-31&to=2026-04-27` returns per-area breakdown. Frontend renders as a bar chart using Recharts.

Comparison shown: actual pct vs `focusBudgetPct` from the area definition.

## 9. Nudge logic

All nudges are in-app only. No push notifications in MVP.

| Trigger | Nudge | Location |
|---|---|---|
| Habit missed 2 consecutive days | Soft inline nudge below habit row | Today — habits card |
| Habit missed 3 consecutive days | Amber background on habit row + prompt | Today — habits card |
| Area no session in 4+ days (expected active day) | Momentum chip turns amber | Today — momentum card |
| Area no session in 7+ days | Momentum chip turns red | Today — momentum card |
| Work block missed 3+ weeknights this week | Pattern note appended to routines card | Today — routines card |
| Rock approaching end of quarter without done status | Rock flagged at_risk | Me tab + weekly review |

Life tasks: **no nudges**. Purely satisfying to complete, never stressful to miss.

Habit nudge calculation runs on the frontend from `habit_logs` data. No separate nudge API needed.
